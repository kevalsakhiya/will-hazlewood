# Database reference

Postgres is the **authoritative store** (RULES §0.2) — Sheets and Drive CSV are derivative views. Anything that affects downstream behaviour must be in Postgres before it's announced.

Six tables across four migrations under [`sql/migrations/`](../sql/migrations/). All migrations are forward-only, idempotent (every `CREATE` uses `IF NOT EXISTS`, every `ALTER` uses `ADD COLUMN IF NOT EXISTS`), and applied via `poetry run python -m broker_scout.tools.migrate`.

## Table summary

| Table | Owner | Lifecycle | Append-only? |
|---|---|---|---|
| [`dld_brokers`](#dld_brokers) | `tools/fetch_dld.py` | Weekly upsert | No (UPSERT on `brn`) |
| [`scrape_runs`](#scrape_runs) | `PostgresPipeline` | One row per spider run | No (UPDATE on close) |
| [`brokers`](#brokers) | `PostgresPipeline` | One row per `(run_id, platform, brn)` | Yes (`ON CONFLICT DO NOTHING`) |
| [`bad_items`](#bad_items) | `PostgresPipeline` | Drained from `spider.bad_items` on close | Yes |
| [`sheet_registry`](#sheet_registry) | `sheets_repo` | One row per `(platform, period)` | No (`is_active` flips) |
| [`alert_log`](#alert_log) | `monitors/actions.py` | One row per chat alert sent | Yes |

## `dld_brokers`

Source of truth for the licensed-broker list. Refreshed weekly by `tools/fetch_dld.py`. The spider iterates `WHERE active = TRUE` to build its DLD seed set every run.

UPSERT on `brn` so DLD field changes (agency moves, expiry dates, awards count, etc.) propagate without losing history of when we first saw them.

| Column | Type | Notes |
|---|---|---|
| `brn` | `TEXT PRIMARY KEY` | DLD `CardNumber` — the regulator's unique broker ID. Cross-referenced everywhere. |
| `office_license_number` | `TEXT` | DLD `LicenseNumber`. Shared across brokers in the same office. |
| `broker_name_en` | `TEXT` | English name from `CardHolderNameEn`. |
| `broker_name_ar` | `TEXT` | Arabic name. |
| `phone` | `TEXT` | Office phone. |
| `mobile` | `TEXT` | Personal mobile. |
| `email` | `TEXT` |  |
| `real_estate_number` | `TEXT` | Office RERA number. Indexed for office-level queries. |
| `office_name_en` | `TEXT` | `OfficeNameEn` — readable agency name. |
| `office_name_ar` | `TEXT` | Arabic version. |
| `card_issue_date` | `DATE` | When the broker's licence was first issued. |
| `card_expiry_date` | `DATE` | When it expires. Filter `>= today` to find currently-valid brokers. |
| `office_issue_date` | `DATE` |  |
| `office_expiry_date` | `DATE` |  |
| `photo_url` | `TEXT` | DLD-hosted broker portrait. |
| `office_logo_url` | `TEXT` |  |
| `card_rank_id` | `INT` | DLD ranking system internal ID. |
| `card_rank` | `TEXT` | Human-readable rank label. |
| `office_rank_id` | `INT` |  |
| `office_rank` | `TEXT` |  |
| `awards_count` | `INT` |  |
| `first_seen_at` | `TIMESTAMPTZ NOT NULL` | When we first saw this BRN. Survives subsequent fetches. |
| `last_seen_at` | `TIMESTAMPTZ NOT NULL` | Updated on every UPSERT. |
| `last_seen_run_id` | `UUID NOT NULL` | The DLD-fetch run that last saw this broker. |

**Indexes**: `(last_seen_run_id)`, `(real_estate_number)`.

**Common query**: brokers in a specific office:
```sql
SELECT brn, broker_name_en
FROM dld_brokers
WHERE real_estate_number = '<RERA number>';
```

## `scrape_runs`

One row per spider run. `PostgresPipeline.spider_opened` opens it; `close_run` finalizes status + counts; `engine_stopped` re-snapshots the final stats blob (post all pipeline flushes).

| Column | Type | Notes |
|---|---|---|
| `run_id` | `UUID PRIMARY KEY` | Generated by `RunIdExtension` at `spider_opened`. Threads through every emitted log line, every broker row, every alert. |
| `spider` | `TEXT NOT NULL` | `agent_spider` (PropertyFinder) or `bayut_spider` (later). |
| `started_at` | `TIMESTAMPTZ NOT NULL DEFAULT now()` | Run start. |
| `finished_at` | `TIMESTAMPTZ` | Run end (NULL while running). |
| `status` | `TEXT NOT NULL DEFAULT 'running'` | `running` → `ok` (clean finish + `finish_reason` in `SUCCESSFUL_REASONS`) or `failed` (anything else). |
| `items_scraped` | `INT` | `item_scraped_count` from Scrapy stats. |
| `items_dropped` | `INT` | `item_dropped_count` (validation rejections + Drop-on-error). |
| `stats` | `JSONB` | Full Scrapy stats blob, post-flush. Every counter the run produced lives here — see [`data-flow.md` §stat-namespaces](data-flow.md#stat-namespaces). |

**Common query**: recent run health:
```sql
SELECT run_id, status, items_scraped, items_dropped,
       stats->>'finish_reason'      AS finish_reason,
       stats->>'log_count/ERROR'    AS errors,
       stats->>'gsheets/flush_failed' AS sheets_failed
FROM scrape_runs
ORDER BY started_at DESC
LIMIT 5;
```

## `brokers`

Append-only — one row per matched/attempted DLD broker per spider run. Even `not_found` and `ambiguous` cases land here as stubs (RULES §0.1, §2.3) so monitor coverage queries are meaningful.

`ON CONFLICT (run_id, platform, brn) DO NOTHING` makes re-runs of the same flush safe.

### Provenance + match metadata

| Column | Type | Notes |
|---|---|---|
| `id` | `BIGSERIAL PRIMARY KEY` | Surrogate. |
| `run_id` | `UUID NOT NULL REFERENCES scrape_runs(run_id)` | FK back to the run. |
| `scrape_date` | `DATE NOT NULL` | UTC date when the run started. Indexed. |
| `platform` | `TEXT NOT NULL` | `propertyfinder` or `bayut`. |
| `brn` | `TEXT` | The broker's PF/Bayut profile BRN. NULL for `not_found` stubs. Indexed. |
| `match_status` | `TEXT NOT NULL DEFAULT 'unknown'` | `exact_brn` / `name_unique` / `name_fuzzy` / `ambiguous` / `not_found` / `unknown`. See [`data-flow.md` §match-status state machine](data-flow.md#match-status-state-machine). |
| `match_confidence` | `NUMERIC` | 0..1. Set by the matching layer. |
| `dld_brn` | `TEXT` | DLD ground-truth BRN. Differs from `brn` only on `match/brn_drift` cases. |
| `dld_broker_name` | `TEXT` | DLD ground-truth name. Useful when `broker_name` is NULL on stubs. |
| `agency_name` | `TEXT` | DLD `OfficeNameEn` — the readable agency label (PF only gives a URL). |

### Identity

| Column | Type | Notes |
|---|---|---|
| `agent_url` | `TEXT` | PF profile URL. |
| `broker_name` | `TEXT` | Name as PF reports it (post-strip). |
| `nationality` | `TEXT` | From `nationality.name` in PF JSON. |
| `agent_specialization` | `TEXT` | "Sales", "Leasing", etc. |
| `experience_since` | `INT` | Year. |
| `whatsapp_response_time` | `INT` | Seconds. Negatives treated as missing. |
| `is_superagent` | `BOOLEAN` | PF's "superagent" flag. |

### Agency

| Column | Type | Notes |
|---|---|---|
| `agency_url` | `TEXT` | PF agency page. |
| `agency_registration_number` | `TEXT` | Agency RERA number from the agency page. |

### Listings

| Column | Type | Notes |
|---|---|---|
| `listings_for_sale` | `INT` |  |
| `listings_for_rent` | `INT` |  |
| `listings_total` | `INT` | Sum of the two above. |
| `listings_with_marketing_spend` | `INT` | Listings flagged premium / featured / spotlight. |
| `average_listing_price_sale` | `NUMERIC` | AED. Aggregated from the listings API. |
| `average_listing_price_rent` | `NUMERIC` |  |
| `average_listing_age_days_sale` | `NUMERIC` | Days since `listed_date`. |
| `average_listing_age_days_rent` | `NUMERIC` |  |
| `most_recent_listing_date_sale` | `DATE` |  |
| `most_recent_listing_date_rent` | `DATE` |  |

### Closed deals (PF "claimed transactions")

| Column | Type | Notes |
|---|---|---|
| `closed_transaction_sale` | `INT` | Count. Zero is a valid value. |
| `closed_transaction_rent` | `INT` |  |
| `closed_deals_total` | `INT` | Sum. NULL only when both sale + rent are missing. |
| `closed_transaction_deal_value` | `NUMERIC` | Total value, AED. |
| `closed_transaction_sale_total_amount` | `NUMERIC` |  |

### Forensics

| Column | Type | Notes |
|---|---|---|
| `raw` | `JSONB` | Full unflattened item — every field the spider produced, exactly as emitted. Used to diff what we extracted vs. what's currently on PF. **Never query into JSONB as if it were structured data** (RULES §13.4); promote to a real column if you need to filter on it. |

**Indexes**: `(brn)`, `(scrape_date)`, `UNIQUE (run_id, platform, brn)`.

**Common queries**:

Match-status distribution for a run:
```sql
SELECT match_status, COUNT(*)
FROM brokers
WHERE run_id = '<run_id>'
GROUP BY match_status
ORDER BY 2 DESC;
```

PF↔DLD BRN drift cases (where `BRNDriftMonitor` would fire):
```sql
SELECT brn, dld_brn, broker_name, dld_broker_name, agency_name
FROM brokers
WHERE run_id = '<run_id>'
  AND brn IS NOT NULL
  AND dld_brn IS NOT NULL
  AND brn != dld_brn;
```

Latest enriched row for a specific broker (across all runs):
```sql
SELECT *
FROM brokers
WHERE brn = '<brn>'
  AND match_status IN ('exact_brn', 'name_unique', 'name_fuzzy')
ORDER BY scrape_date DESC
LIMIT 1;
```

## `bad_items`

Validation drops. Drained from `spider.bad_items` (in-memory list set by `RunIdExtension`, populated by `ValidationPipeline`) into this table on `spider_closed`.

| Column | Type | Notes |
|---|---|---|
| `id` | `BIGSERIAL PRIMARY KEY` | Surrogate. |
| `run_id` | `UUID` | Which run. Indexed. |
| `platform` | `TEXT` |  |
| `reason` | `TEXT` | The validation failure message — usually the pydantic error path + value. |
| `payload` | `JSONB` | The full item the spider tried to emit. Inspect to see the exact field that broke. |
| `created_at` | `TIMESTAMPTZ NOT NULL DEFAULT now()` |  |

**Indexes**: `(run_id)`.

**Common query**: what failed on the latest run:
```sql
SELECT reason, payload->>'brn' AS brn, payload->>'broker_name' AS broker_name
FROM bad_items
WHERE run_id = (SELECT run_id FROM scrape_runs ORDER BY started_at DESC LIMIT 1)
LIMIT 20;
```

## `sheet_registry`

Drives the monthly Sheets rotation. The Sheets pipeline calls `get_or_create_active_sheet(platform)` on `spider_opened` — if no active row exists for `(platform, current_period)`, the pipeline copies the template via Drive API and inserts a new row here. Older periods get `is_active = FALSE` so replay/forensics can still find them.

| Column | Type | Notes |
|---|---|---|
| `id` | `BIGSERIAL PRIMARY KEY` | Surrogate. |
| `platform` | `TEXT NOT NULL` | `propertyfinder` / `bayut`. |
| `period` | `TEXT NOT NULL` | Month key, `YYYY-MM` format. |
| `sheet_id` | `TEXT NOT NULL` | Google Sheets file ID for the active spreadsheet of that period. |
| `is_active` | `BOOLEAN NOT NULL DEFAULT TRUE` | One TRUE per platform at any time. |
| `created_at` | `TIMESTAMPTZ NOT NULL DEFAULT now()` |  |

**Indexes**: `(platform, is_active)`, `UNIQUE (platform, period)`.

**Common queries**:

Which spreadsheet is the spider writing to right now?
```sql
SELECT platform, sheet_id
FROM sheet_registry
WHERE is_active = TRUE
ORDER BY platform;
```

Find a historical period:
```sql
SELECT sheet_id
FROM sheet_registry
WHERE platform = 'propertyfinder' AND period = '2026-04';
```

The race-safe insert is `ON CONFLICT (platform, period) DO NOTHING` — if two processes try to create the same period concurrently, one wins and the other detects 0 rowcount and re-selects (RULES §14.2).

## `alert_log`

Every chat alert (Discord / Google Chat) sent by `monitors/actions.py` lands here. Append-only. Used by `SendCriticalChatAlertAction` to dedupe within a 30-minute window — if the same `(level, title)` was already sent recently, the action no-ops to prevent storm-spam.

| Column | Type | Notes |
|---|---|---|
| `id` | `BIGSERIAL PRIMARY KEY` |  |
| `run_id` | `UUID` | Which run triggered the alert. NULL if pre-`spider_opened`. |
| `level` | `TEXT` | `warning` or `critical`. |
| `title` | `TEXT` | The card title (e.g. `agent_spider — CRITICAL`, `Circuit breaker tripped`). Dedupe key. |
| `body` | `TEXT` | Full card body that was sent. |
| `sent_at` | `TIMESTAMPTZ NOT NULL DEFAULT now()` |  |

**Common queries**:

Last 10 alerts:
```sql
SELECT sent_at, level, title
FROM alert_log
ORDER BY sent_at DESC
LIMIT 10;
```

Has a critical title fired recently? (the dedupe lookup):
```sql
SELECT 1 FROM alert_log
WHERE level = 'critical'
  AND title = 'Circuit breaker tripped'
  AND sent_at > now() - interval '30 minutes';
```

## Cross-table relationships

```
dld_brokers (brn PK)
   │
   │  spider iterates dld_brokers and emits one item per row
   ▼
brokers ──── run_id FK ────► scrape_runs
   │
   ├── (run_id, platform, brn) UNIQUE — re-runs idempotent
   │
   └── dld_brn references the dld_brokers.brn the row was seeded from
       (no FK constraint — DLD rows can be deleted without cascading)

bad_items ──── run_id (no FK) — best-effort, may exist without a
                                scrape_runs row if the run died early

sheet_registry — independent of run history; only used by sheets_repo
                 to resolve the active spreadsheet per period

alert_log — independent of run history; queried by alerts dedupe
```

The deliberate choices:
- **`brokers.run_id` is a real FK** — referential integrity matters; orphaned broker rows would corrupt monitor queries.
- **`bad_items.run_id` is NOT a FK** — a spider crash before `open_run` may still leave a partial item buffer. Capturing those is more valuable than enforcing the FK.
- **`alert_log.run_id` is NOT a FK** — alerts can fire from contexts where no `scrape_runs` row exists yet (e.g. a process that dies before `spider_opened`).

## Schema evolution

Adding fields, renaming columns, or dropping unused ones — see RULES §13.5 and the per-step checklist in [`data-flow.md` §schema-evolution-rules](data-flow.md#schema-evolution-rules).

Short version:
- **Adding a nullable column** is fine. New migration with `ADD COLUMN IF NOT EXISTS`.
- **Adding a NOT NULL column** needs a default OR a backfill migration in two parts.
- **Renaming a column** = add new + drop old in two separate migrations. Never rename in place if any committed code reads the old name.
- **Numbered, forward-only.** Never edit a committed migration.

## Connection + tooling

- Pool: `common/db.py` — lazy psycopg connection pool, single instance per process. Read DSN from `POSTGRES_*` env vars.
- Repo modules: `common/dld_repo.py`, `common/brokers_repo.py`, `common/sheets_repo.py` — every Postgres-touching call goes through one of these (RULES §5.1). No module owns its own pool or builds raw SQL outside its repo.
- Identifier safety: `psycopg.sql.Identifier` is used anywhere a column name comes from a variable (RULES §5.3) — relevant inside `matched_field_coverage` for the per-tier dynamic coverage queries.

To open a psql shell against the dev DB:

```bash
docker exec -it broker_scout_postgres psql -U root -d broker_scout
```
